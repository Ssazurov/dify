# Общие сведения

Solar SafeInspect имеет **Application Programming Interface** (далее – **API управления**, **API**) на основе **REST**, которое позволяет запрашивать, создавать, обновлять и удалять пользователей и учетные данные в Solar SafeInspect с помощью **API**. **API** **управления** – это инструмент, который позволяет интегрировать Solar SafeInspect и, например, программное обеспечение для обеспечения доступа пользователей. При использовании программного обеспечения для обеспечения доступа пользователей, работающих вне Solar SafeInspect, можно, например, предоставить пользователям права на определенные хосты, распространять ключи, предоставленные учетным записям пользователей, или обновлять пароли пользователей. **API** позволяет автоматизировать различные операции.

Ниже приведены примеры использования **API** с использованием примера реализации для клиента **API**. Есть также возможность использовать пример **API – клиента** как библиотеку **Python**, возможность его редактирования и использования в качестве основы для своей собственной реализации.

# Включение API управления

Чтобы читать или записывать информацию через **API**, требуется:

1. **Локальная учетная запись администратора** с соответствующими правами, связанными с API (вы можете использовать существующую учетную запись администратора или создать новую);
2. *Дополнительно*: **Сертификат клиента для аутентификации**. **API** поддерживает аутентификацию клиента на основе сертификатов **X.509** в качестве альтернативы парольной аутентификации.
# Использование PKCS#12 для API управления доступом

Получить доступ к **API управления** с помощью сертификата **CA** Solar SafeInspect и контейнера **PKCS #12**, включая закрытый ключ и его сертификат **X.509**, можно следующим образом:

1. Перейдите на вкладку **Настройки > Администраторы > Локальные учетные записи**.
2. Выбрать учетную запись администратора (Рис. 1).
3. В блоке **Сертификаты администратора** нажать кнопку **Сгенерировать PKCS #12** (Рис. 2) для того, чтобы загрузить **контейнер**.
![](Solar_SafeInspectt_2.4.10_Инструкция_по_работе_с_API_media\image001.png)

Рис. 1. Клиент API

![](Solar_SafeInspectt_2.4.10_Инструкция_по_работе_с_API_media\image002.png)

Рис. 2. Генерация PKCS#12

4. Перейдите на вкладку **Настройки > Управление сертификатами > Проверка сертификатов** и нажмите **Сертификат внутреннего центра сертификации** (Рис. 3) для того, чтобы загрузить внутренний сертификат CA Solar SafeInspect. Сертификат CA необходим для проверки сертификата сервера Solar SafeInspect. В открывшемся окне нажмите **Скачать сертификат**.
![](Solar_SafeInspectt_2.4.10_Инструкция_по_работе_с_API_media\image003.png)

Рис. 3. Сертификат внутреннего центра сертификации

5. Извлеките закрытый ключ и его сертификат из контейнера **PKCS #12**. Например, используйте **OpenSSL**, следующим образом:
| $ openssl pkcs12 -in <username>-cert.p12 -out <username>-key.pem -nocerts -nodes |
|---|
| $ openssl pkcs12 -in <username>-cert.p12 -out <username>-crt.pem -clcerts -nokeys |

6. Сконвертируйте сертификат внутреннего CA Solar SafeInspect в **PEM – формат** следующей командой:
| $ openssl x509 -inform DES -in internal-ca.crt -out internal-ca.pem -text |
|---|

7. Теперь можно **проверить доступ к API управления**, например, с помощью **curl **следующим образом:
| $ curl -E <username>-crt.pem --key <username>-key.pem --cacert internal-ca.pem https://<SafeInspect-ip-address>/api/ |
|---|

**API** вернет список конечных точек, к которым вы можете получить доступ:

| $ ["/", "status/", "authkeys/", "authkeys/usergroups/", "authkeys/hostgroups/", "authkeys/userproviders/", "usermappings/", "usermappings/usergroups/", "usermappings/hostgroups/", "usermappings/userproviders/", "hostkeys/", "users/", "users/authorized_keys/", "users/tokens/", "users/auth/", "enroll_api_certificate/", "revoke_api_certificate", "client_library/", "help/"] |
|---|

Можно использовать **curl**, чтобы увидеть, например, **статус API**, следующим образом:

| $ curl -E <username>-crt.pem --key <username>-key.pem --cacert internal-ca.pem https://<SafeInspect-ip-address>/api/status |
|---|

**API** вернет следующую **информацию о статусе**:

| {"status": true, "current_time": "2016-03-03T13:34:07.957+00:00", "version": "2.0.0.364", "fips": false, "critical_disk_utilization": false, "statistics": {"connections": 493, "channels": 321, "open_connections": 0, "open_channels": 0}, "started_at": "2016-03-03T05:51:10.000+00:00", "pid": 45278, "api_version": 1, "user": {"certificate_fingerprint": "9767730d76afc23a4e070c621a60c17ccc7f3850", "certificate_issued_at": "2016-03-03T09:56:28.000+00:00", "name": "admin", "certificate_serial": "6586", "certificate_not_after": "2021-03-02T12:56:28.000+00:00"}} |

## Типы операций

Операции выполняются с использованием методов **HTTP**. Методы **HTTP** и их значения представлены в таблице (Табл. 1). Для выполнения операции необходимо использовать параметры из столбца **Значение**.

Табл. 1. Методы HTTP

| Метод HTTP | Значение | Выполняемая операция |
|---|---|---|
| GET | search | Поиск и вывод содержимого конфигурации |
| PUT | add | Добавление записи, например, учетных данных пользователя в конфигурацию |
| POST | update | Изменение значений параметров |
| DELETE | remove | Удаление записи, например, пользователей |

Для определения типа операции используется параметр **'method'**. В данном случае доступны значения **'search'**, **'add'**, **'update'** и **'remove'**.

Примечание:

> В **Windows** для включения строк в командной строке требуются кавычки (") вместо апострофов ('). Если в строках есть еще строки, они должны быть заключены с экранированными кавычками (\").

## Параметры операций

Запросы могут включать в себя список параметров, некоторые из них являются необязательными. Параметры могут задаваться непосредственно в строке ('?par=val;...') и формате **JSON**.

## Результаты операций

### HTTP коды состояния

При работе с **API** возможны следующие коды состояния:

- **200** – код состояния успешных операций;
- **400 – **код информирует о неудачном запросе (например, Unknown parameters '[u'value']');
- **403** – код информирует о неудачной аутентификации или авторизации;
- **404 **– код информирует о том, что не удается найти данные по запросу (например, Path not found);
- **500 **– код информирует о внутренней ошибке сервера (например, 500 Internal Server Error).
### HTTP тело ответа

Все ответы имеют формат **JSON**. Общее логическое поле **'status'** информирует об успешном или неудачном запросе.

### Ответ о неудачном запросе

В случае неудачи поле **'result'** содержит значение **'error'**, в котором указано сообщение об ошибке.

Ответ о неудаче может содержать также и другие значения, например, **'details'**, которое содержит детальную информацию или информацию о том, как исправить возникшую ошибку.

### Ответ об успешном запросе

В ответе об успешном запросе обязательно содержатся два поля:

**'status'**, имеющее значение true;

фактическое поле ответа **'result'**, которое зависит от выполняемой операции.

Кроме того, все результаты содержат поле **'api_version'**, которое информирует о версии **API**.

## Используемые форматы данных

Все значения форматируются в соответствии с **ISO 8601**.

Значение **Дата и Время** представлено как **YYYY – MM – DDTHH:MM:SS**.

Бинарные данные, такие как сертификаты, публичные и приватные ключи представлены в соответствующих им форматах (например, **PEM – формат** или **base64**).

## Простые запросы

Помимо параметров подключения, запрос к **API** содержит команду и ее опциональные параметры. Существуют такие команды, как **'status'**, **'authkeys'** или **'help'**.

Пример:

При использовании команды **status**, будет запрошен статус сервера.

Команда **'help'** выводит содержимое документации на **API** управления Solar SafeInspect.

Параметры определяются как **'параметр=значение'**. Если параметр содержит внутри значение блока, он может быть представлен как **'values.user=test1'**.

Параметры можно задавать с помощью специальных команд **–– par** (или **– p**).

Если значение параметра имеет сложный вид, например, список или набор списков, то его можно задать с помощью команды **–– par – json** (или **– j**).

Пример:

--par-json values.attributes='{"email": "user@host.com"}'

значение параметра будет обработано в формате **JSON**.

## Расширенные параметры

Помимо назначения параметров в командной строке, их можно считать из файла, переменной окружения или задать отдельной командой.

Воспользуйтесь примером добавления сопоставления, где **comment** – описание сопоставления, **password** – пароль конечного пользователя, который читается из файла **password .file**:

| $ python3 api_client.py --settings-file=admin-api-config.dat authkeys --par-file password=password.file --par-command comment="echo 12345" --json '{"method": "add", "gw_user": "123","address": "10.2.1.45","port": "23","protocol": "secsh","user": "123","policy": {}}' |
|---|

**–– par – fd** и **–– par – env** работают аналогично: для **–– par – env** требуется указать переменную окружения локальную, а для **–– par – fd** файловый дескриптор файла, с которого будет читаться информация (например, файловый дескриптор файла **password.file**).

Этот способ позволяет обезопасить передаваемые данные, не указывая их непосредственно в командной строке. Весь блок параметров можно задать с помощью команд **–– json** и **–– json – file**, последняя позволит считать их из файла.

### Параметр csv–file

Используя параметр **'par – csv – list'**, можно задать соответствие содержимого файла определенным параметрам.

Пример:

'--par-csv-list переменная=файл:col1,col2'

в данном синтаксисе:

**'переменная'** – список параметров (например, **'entries'**);

**'файл'** – имя файла с загружаемыми значениями параметров;

**col1**, **col2** – параметры, которым присваиваются значения.

Если **CSV – файл** содержит столбцы, которые необходимо пропустить, это можно указать, используя параметр **'ignore'**.

Пример:

Файл **'csv.data'** содержит:

| 1;123.10.0.7;ret;ignore 3;210.1.101.0/24;tet;Internal network 3;210.1.102.1;pet;kawabanga |
|---|

Следующая команда считает эти данные из файла, задаст три элемента групп хостов за одну операцию, при этом нужно чтобы были созданы группы с **id** 1 и 3 (при выполнении аналогичного примера):

| $ python3 api_client.py --settings-file=admin-api-config.dat hostgroups/entries method=add --par-csv-list entries=csv.data:group_id,address,name,desc |
|---|

### Импорт csv

Для импорта данных из **CSV** можно использовать команду **–– csv – data**, которая указывает соответствие содержимого файла и параметров **API**.

Пример:

Файл **'csv.data'** содержит:

matti;ug1;df03kjw8

test1;ug1;4kdm6lls

user1;ug2;dlmfj50d

Команда:

| authkeys method=add hostgroup=hg1 comment='csv import' --csv-file csv.data --csv-data user,usergroup,password |
|---|

добавит три элемента в **authkeys** построчно из файла **'csv.data'** с общими параметрами **hostgroup** и **comment**.

Если **CSV – файл** содержит столбцы, которые необходимо пропустить, это можно указать, используя параметр **'ignore'** команды **–– csv – data**.

### Форматирование вывода

По умолчанию вывод представлен в формате **JSON**. Для вывода конкретного параметра необходимо использовать команду ––**result**. Если в команде ––**result** задать несколько параметров, они будут выведены с разделителем.

Пример:

# ... status --result current_time 2016-12-09T11:29:23.065+03:00

Вывод можно представить в виде таблицы, используя команду ––**columns**, при этом необходимо выполнить следующие условия:

1. после команды ––**columns** необходимо наличие параметра <**имена_столбцов**>;
2. имена столбцов необходимо указать через запятую, без пробела.
Пример:

| # ... users --columns user_id,username user_id \| username \| ----------------------- 1 user3 2 user1 3 user2 4 user4 5 user5 6 user6 |
|---|

### Добавление SafeInspect в среду с существующими ключами

Ввести SafeInspect в среду, в которой уже есть открытые и закрытые ключи можно, запустив API–клиент со следующими командами для создания локального пользователя, добавления существующего открытого ключа в Менеджер SafeInspect для аутентификации **Пользователь – Коллектор** и создания пары ключей для аутентификации **Коллектор – Целевой сервер**.

1. Создайте локального пользователя без пароля, указав следующую команду:
| $ python3 api_client.py --settings-file <имя-файла>.dat users method=add username=<имя-пользователя> |
|---|

2. **Добавьте существующий открытый ключ** (в формате **OpenSSH**) в существующую локальную учетную запись пользователя, указав следующую команду:
| $ python3 api_client.py --settings-file <имя-файла>.dat users/authorized_keys method=add username=<имя-пользователя> --par-file key=</абсолютный/путь/до/ключа> |
|---|

3. **Создайте учетные данные закрытого ключа для локального пользователя**, который затем может получить доступ к целевому хосту в качестве целевого пользователя. Пара ключей генерируется ключом **generate_key=true**. Закрытый ключ будет сохранен в Менеджере SafeInspect. Открытый ключ загружается на целевой сервер с использованием пароля целевого пользователя, указанного в параметре **upload.password = <пароль – пользователя – целевой – системы>**. Используйте **upload.hostkey = ignore**, чтобы отключить проверку ключа хоста. В этом случае, если ключ хоста неизвестен, ключ будет автоматически сохранен в Менеджере SafeInspect. Запустите клиент API следующим образом:
| $ python3 api_client.py --settings-file <имя-файла>.dat usermappins method=add gw_user=<имя-пользователя> address=<адрес-целевого-сервера> user=<пользователь-целевой-системы> generate_key=true upload.password=<пароль-пользователя-целевой-системы> upload.hostkey=ignore |
|---|

### Создание ключевой пары для аутентификации с открытым ключом

В этом примере генерируется пара ключей, чтобы включить аутентификацию с открытым ключом между SafeInspect и целевым сервером. Закрытый ключ будет сохранен в Менеджере SafeInspect, и вам необходимо вручную авторизовать открытый ключ на целевом сервере.

На примере имеется локальная группа пользователей, называемая **example – group**, целевой хост, находящийся по адресу **192.0.2.101**, целевой пользователь (**testuser**), и файл настроек (**admin – api – config.dat**). Пара ключей генерируется с помощью параметра **generate_key = true**.

Запустите **клиент** **API** со следующими параметрами:

| $ python3 api_client.py --settings-file admin-api-config.dat usermappings method=add usergroup=example-group address=192.168.2.101 user=testuser generate_key=true |

### Политики токена доступа

Политики описывают ограничения и директивы, которые влияют на токен доступа или ключ авторизации.

Поддерживаются следующие директивы (Табл. 30).

Табл. 30. Директивы, влияющие на токен доступа

| Параметр | Описание |
|---|---|
| not–before(YYYY–MM–DD HH:MM:SS) | Токен не может быть использован до заданного времени |
| not–after(YYYY–MM–DD HH:MM:SS) | Токен не может быть использован после заданного времени |
| relative–validity(СЕКУНД) | Токен может быть использован в течение заданных секунд с момента начала его использования. Можно комбинировать с not–before и not–after |
| max–use–count(N) | Может быть использован **N** раз |
| time–window(pattern(PAT) tz(TZ)) | Определяет окно использования токена: **PAT** – шаблон в формате **'SEC MIN HOUR WEEKDAY MONTHDAY YEARDAY MONTH YEAR'**, где каждый параметр указан явно или звездочкой ' '. Параметры могут быть заданы как диапазон. Пример: **'8:00 – 15:00 mo – fr * * * *'** определяет окно использования с 8:00 до 15:00 каждый день с понедельника по пятницу. Необязательный параметр временной зоны **TZ** задается в формате абсолютного значения +0300 или в формате имени **Europe/Moscow**. Если параметр **TZ** не задан, временной зоной задается по умолчанию **UTC**. Можно задать несколько временных окон, и они будут объединены |
| limit–session–time() | Срок истечения токена доступа или временное окно, которое не только указывает период действительности токена, но и устанавливает ограничение продолжительности сессии |
| max–session–time(СЕКУНД) | Определяет лимит сессии в секундах. Если помимо лимита сессии заданы временное окно или срок истечения, будет выбрано меньшее значение |
| hostgroup() | Определяет группу хостов, на доступ к которой действует токен. Допустимо задание нескольких групп хостов |
| address() | Определяет адрес, на доступ к которому действует токен. Допустимо задание нескольких адресов |
| relay–password(boolean) | Определяет будет ли пароль от входящего соединения передан на исходящее |
| allow–relay–auth(boolean) | Определяет в случае ошибки ретранслировать аутентификационную информацию или нет (при использовании сопоставления пользователей). Пример: **Строка политики not – after(2016 – 12 – 01 00:00:00) relative – validity(600) time – window(pattern(* * 8 – 15 mo – fr * * * *)) max – session – time(3600)** определяет, что токен доступа может быть использован в течение 10 минут с момента активации, но не позже 1 декабря 2016 года, для доступа в рабочие дни с 8 до 15 на сессию, длительностью 1 час |

Создавать и обновлять правила, в том числе с **Коллектором и Обработчиком**, можно с помощью следующих примеров. В них сообщается, как создавать и обновлять правила, используя пример **клиента** **API** и **cURL**.

### Создание стандартного правила без Коллекторов

Для того, чтобы создать стандартное правило **без Коллекторов ,** необходимо выполнить следующую команду:

$ python3 api_client.py --settings-file=admin-api-config.dat rules --par "method=add" --json '{"rule_settings": {"name": "<название-правила>", "rule_type": "ssh","to_port": 22, "add_criteria": {"bastion_listener": [{"port": 22, "interface": "0.0.0.0"}]}}}'

### Создание правила с сопоставлением пользовательских групп

Для того, чтобы создать правило **с сопоставлением пользовательских групп**, необходимо выполнить следующую команду:

curl -d '{"method": "add", "audit_actions": {"to_user_group": [{"name":"<название-группы-для-сопоставления>", "type": "local"}]}, "rule_settings": {"rule_type": "ssh","user_authentication_from_client_to_hound": "local", "user_authentication_from_hound_to_target": "mapped", "name": "<название-правила>", "add_criteria": {"to_port": [22,23,24]}}}' -H "Content-Type: application/json" https:// <адрес-целевой-системы>/api/rules/ --user <пользователь-целевой-системы>:< пароль-пользователя-целевой-системы> -k -vv

В результате команды в правило <**название – правила**> будут добавлены порты 22 – 24, выставлено сопоставление учетных данных, добавлено сопоставление пользовательской группы **<название – группы – для – сопоставления>.**

### Обновление правила (добавление Коллектора)

Для того, чтобы обновить правило (**добавить Коллектор**) необходимо выполнить следующую команду:

$ python3 api_client.py --settings-file=admin-api-config.dat rules --par "method=update" --json '{"rule_settings": {"name": "<название-правила>", "rule_type": "ssh","to_port": 22, "hounds": ["OUT"], "add_criteria": {}}}'

В данном примере **"hounds": ["OUT"]** добавит **все **Коллекторы. Для добавления конкретного коллектора необходимо указать **"hounds": ["<название – коллектора>"].**

Также можно использовать **cURL**, чтобы обновить правило (**добавить Коллектор**), следующим образом:

$ curl -d '{"method": "update", "rule_settings": {"rule_type": "ssh", "name": "<название-правила>", "hounds": ["OUT"],"add_criteria": {"bastion_listener": [{"port": 22, "interface": "0.0.0.0"}]}}}' -H "Content-Type: application/json" https://<адрес-целевой-системы>/api/rules/ --user <пользователь-целевой-системы>:< пароль-пользователя-целевой-системы> -k -vv

В данном примере **"hounds": ["OUT"]** добавит **все **Коллекторы. Для добавления конкретного коллектора необходимо указать **"hounds": ["<название – коллектора>"].**

### Создание правила с Коллектором и Обработчиком

Для того, чтобы создать правило **с Коллектором и Обработчиком** необходимо выполнить следующую команду:

$ python3 api_client.py --settings-file=admin-api-config.dat rules --par "method=add" --json '{"rule_settings": {"name": "<название-правила>", "rule_type": "ssh","to_port": 22, "hounds": ["OUT"], "add_criteria": {"bastion_listener": [{"port": 22, "interface": "0.0.0.0"}]}}}'

В данном примере **"hounds": ["OUT"]** добавит **все **Коллекторы. Для добавления конкретного коллектора необходимо указать **"hounds": ["<название – коллектора>"].**

Вы можете использовать **cURL**, чтобы создать правило с **Коллектором и Обработчиком**, следующим образом:

$ curl -d '{"method": "add", "rule_settings": {"rule_type": "ssh", "name": "<название-правила>", "hounds": ["OUT"], "add_criteria": {"bastion_listener": [{"port": 22, "interface": "0.0.0.0"}]}}}' -H "Content-Type: application/json" https:// <адрес-целевой-системы>/api/rules/ --user <пользователь-целевой-системы>:< пароль-пользователя-целевой-системы> -k -vv

В данном примере **"hounds": ["OUT"]** добавит **все **Коллекторы. Для добавления конкретного коллектора необходимо указать **"hounds": ["<название – коллектора>"].**

**П оследние версии** **cURL** позволяют делать короткую запись через флаг **–– json**, в данном случае можно не указывать **Content – Type** через **– H**, поскольку **cURL** его сформирует автоматически как **JSON**, например:

$ curl --json '{"method": "add", "rule_settings": {"rule_type": "ssh", "name": "short_api_rule", "hounds": ["OUT"], "add_criteria": {"bastion_listener": [{"port": 22, "interface": "0.0.0.0"}]}}}' https:// <адрес-целевой-системы>/api/rules/ --user <пользователь-целевой-системы>:< пароль-пользователя-целевой-системы> -k -vv

В данном примере **"hounds": ["OUT"]** добавит **все **Коллекторы. Для добавления конкретного коллектора необходимо указать **"hounds": ["<название – коллектора>"].**

## Работа с коллекторами

### Получение информации о Коллекторе

Для того, чтобы получить информацию о Коллекторе, необходимо выполнить команду:

$ curl -v -H "Content-Type: application/json" --data '{"method": "search"}' --user <пользователь-целевой-системы>:< пароль-пользователя-целевой-системы> https:// <адрес-целевой-системы>/api/api_collectors/ -k

В ответ **клиент API** возвращает следующую информацию о Коллекторах.

1. При наличии **внешнего Коллектора **информация будет выглядеть следующим образом:
| { "status": true, "result": [ { "public_ip": "10.2.1.10", "connected": false, "mode": "bastion", "name": "OUT" }, { "public_ip": "10.2.1.198", "connected": true, "mode": "bastion", "name": "Cafall" } ], "api_version": 1 } |
|---|

2. При наличии **кластера Коллекторов **информация будет выглядеть следующим образом:
| { "status": true, "result": [ { "public_ip": "10.2.1.10", "connected": false, "mode": "router", "name": "OUT" }, { "public_ip": "10.2.1.13", "cluster_ip": "10.2.1.12", "connected": false, "mode": "bastion", "name": "Cluster_backup" }, { "public_ip": "10.2.1.11", "cluster_ip": "10.2.1.12", "connected": false, "mode": "bastion", "name": "Cluster" }, { "public_ip": "10.2.1.198", "connected": true, "mode": "bastion", "name": "Cafall" } ], "api_version": 1 } |
|---|

Команда **api_collectors** выводит список коллекторов. Вывод содержит следующие значения, описанные в таблице (Табл. 65).

Табл. 65. Значения вывода списка Коллекторов

| Параметр | Описание |
|---|---|
| ip(string) | IP коллектора |
| mode(string) | Режим работы коллектора. Возможны три значения: 1. Бастион. 2. Режим маршрутизатора. 3. Режим моста. |
| connected(boolean) | Коллектор включен или выключен |

С помощью метода **search** существует возможность получения следующих полей в правилах и обработчиках:

- To port;
- Bastion listener;
- Если аутентификация через **LDAP**: **LDAP server (address, port)**.
Пример получения полей **для правил**:

curl -d '{"method": "search"}' -H "Content-Type: application/json" https://<адрес-целевой-системы>/api/rules/ --user <пользователь-целевой-системы>:<пароль-пользователя-целевой-системы> -k -vv

Пример получения полей **для обработчиков**:

curl -d '{"method": "search"}' -H "Content-Type: application/json" https://<адрес-целевой-системы>/api/listeners/ --user <пользователь-целевой-системы>:<пароль-пользователя-целевой-системы> -k -vv

### Добавление Обработчиков к Коллектору

Для добавления Обработчика к Коллектору необходимо использовать метод **'add':**

| $ python3 api_client.py --settings-file=admin-api-config.dat listeners --json ' { "method": "add", "listeners": [ { "port": 27, "interface": "0.0.0.0", "type": "telnet", "hound": "Cafall" }, { "port": 28, "interface": "0.0.0.0", "type": "http", "hound": "Cafall" } ] }' |
|---|

| { "result": [ { "id": 33, "interface": "0.0.0.0", "port": 27, "config_id": 73, "creator": "admin" }, { "id": 1, "interface": "0.0.0.0", "port": 28, "config_id": 73, "creator": "admin" } ], "need_apply": true, "success": "OK" } |
|---|
