## Приложение Б. Типовые сценарии политик доступа

Ниже приведены примеры настройки политик для распространенных бизнес-сценариев.

### Сценарий 1: «Стандартный сотрудник»
Для доступа к внутренним сервисам (CRM, ERP, почта) в рабочее время.
- **Время доступа:** Пн-Пт, 09:00–18:00.
- **Протоколы:** HTTP/HTTPS, RDP.
- **Ограничения:**
  - Буфер обмена: Включен (текст).
  - Передача файлов: Запрещена.
  - Запись сессии: Только скриншоты (при активности).
- **Длительность сессии:** До 8 часов.

### Сценарий 2: «Внешний подрядчик»
Для временного доступа к тестовым стендам с максимальным контролем.
- **Время доступа:** По расписанию (указывается в тикете).
- **Протоколы:** RDP, SSH.
- **Ограничения:**
  - Буфер обмена: Запрещен.
  - Передача файлов: Запрещена.
  - Запись сессии: Полная видеозапись (SafeInspect).
  - Водяной знак: Включен (логин пользователя).
- **Длительность сессии:** Не более 4 часов.
- **Требование:** Обязательное согласование администратором.

### Сценарий 3: «Аварийный доступ (Break-glass)»
Для экстренного устранения критических инцидентов.
- **Время доступа:** Без ограничений (24/7).
- **Протоколы:** Все доступные.
- **Ограничения:**
  - Буфер обмена: Включен.
  - Передача файлов: Включена.
  - Запись сессии: Полная видеозапись + логирование клавиш.
- **Длительность сессии:** Не более 2 часов (автоматическое завершение).
- **Требование:** Двухфакторная аутентификация (MFA) обязательна.

## Приложение В. Глоссарий

| Термин | Описание |
|---|---|
| **SSO (Single Sign-On)** | Единая система аутентификации, позволяющая пользователю войти в несколько приложений с использованием одного набора учетных данных. |
| **RDP (Remote Desktop Protocol)** | Протокол Microsoft для удаленного управления рабочим столом. |
| **VNC (Virtual Network Computing)** | Протокол графического управления удаленным компьютером, независимый от ОС. |
| **LDAP (Lightweight Directory Access Protocol)** | Протокол для доступа к службам каталогов (например, Active Directory). |
| **SAML (Security Assertion Markup Language)** | Стандарт обмена данными аутентификации и авторизации между сторонами. |
| **MFA (Multi-Factor Authentication)** | Многофакторная аутентификация, требующая два или более способа подтверждения личности. |
| **Zero Trust** | Модель безопасности, предполагающая, что угрозы могут находиться как за пределами, так и внутри сети. |
| **Reverse Proxy** | Сервер, который принимает запросы от клиентов и перенаправляет их на внутренние серверы, скрывая их от внешнего доступа. |
| **Session Recording** | Функция записи действий пользователя в рамках удаленной сессии для аудита. |

## Приложение Г. API Reference (Краткий справочник)

SafeConnect предоставляет REST API для автоматизации управления. Базовый URL: `https://<host>/api/v1/`.

### Основные эндпоинты:

1. **Аутентификация**
   - `POST /auth/login` — получение токена доступа.
   - `POST /auth/logout` — завершение сессии API.

2. **Управление ресурсами**
   - `GET /resources` — список всех ресурсов.
   - `POST /resources` — создание нового ресурса.
   - `PUT /resources/{id}` — обновление параметров ресурса.
   - `DELETE /resources/{id}` — удаление ресурса.

3. **Управление пользователями**
   - `GET /users` — список пользователей.
   - `POST /users` — создание пользователя.
   - `PUT /users/{id}/status` — блокировка/разблокировка.

4. **Сессии**
   - `GET /sessions/active` — список активных сессий.
   - `POST /sessions/{id}/terminate` — принудительное завершение сессии.

> **Примечание:** Для доступа к API требуется роль «Суперадминистратор» или выделенная роль «API Service Account».

## Приложение Д. Системные требования (Детализация)

### 1. Серверная часть (SafeConnect Server)
- **Операционная система:**
  - Windows Server 2016/2019/2022 (Standard/Datacenter);
  - Linux: CentOS 7/8, Ubuntu 20.04/22.04 LTS, Debian 11/12.
- **Процессор:** Минимум 4 ядра (рекомендуется 8+ для нагрузок >500 сессий).
- **Оперативная память:** Минимум 8 ГБ (рекомендуется 16 ГБ+).
- **Диск:** SSD, минимум 50 ГБ свободного места (для логов и БД).
- **Сеть:**
  - Порт `443` (HTTPS) — вход для пользователей;
  - Порт `80` (HTTP) — редирект на HTTPS;
  - Порты `3389`, `22`, `5900` — исходящие для подключения к ресурсам (настраиваются).

### 2. Клиентская часть (Браузер)
- **Поддерживаемые браузеры:** Chrome 90+, Firefox 88+, Edge 90+, Safari 14+.
- **Требования:** Включенный JavaScript, поддержка WebSockets.
- **Плагин:** Не требуется (полностью веб-интерфейс).

## Приложение Е. Лицензирование

SafeConnect поставляется в следующих редакциях:

| Редакция | Количество пользователей | Интеграции | Запись сессий | Поддержка |
|---|---|---|---|---|
| **Starter** | До 50 | LDAP | Нет | Email |
| **Professional** | До 500 | LDAP, RADIUS, SAML | Скриншоты | Телефон/Email |
| **Enterprise** | Без ограничений | Все (вкл. SafeInspect) | Полная запись | 24/7 SLA |

> **Активация:** Лицензионный ключ вводится в разделе **Настройки → Лицензия**. Ключ привязывается к MAC-адресу сервера или IP-адресу.
